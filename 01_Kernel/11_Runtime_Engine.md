# Runtime Engine

## Purpose

The Runtime Engine defines how AI Architect OS operates during a real learning session.

It connects every major component of the Operating System into one continuous workflow.

The Runtime Engine does not store knowledge.

It executes knowledge.

---

## Primary Mission

Execute one complete learning cycle from session start to session completion while preserving learning continuity.

---

## Runtime Flow

Every learning session follows this execution order:

1. Load Operating System
2. Reconstruct Current State
3. Generate Dynamic Session Prompt
4. Start AI Session
5. Teach & Mentor
6. Verify Competency (if necessary)
7. Generate Session Update
8. Update Operating System
9. Prepare Next Session

---

## Step 1 — Load Operating System

Load all information required for the current learning session.

This includes:

- Kernel
- Curriculum
- Daily
- Weekly
- Monthly
- Projects
- Research
- GitHub
- BlackBook
- Reviews
- Resources

---

## Step 2 — Reconstruct Current State

The Context Engine determines:

- Current Sprint
- Current Curriculum Domain
- Current Level
- Current Topic
- Current Priorities
- Available Study Time
- Active Projects
- Current Constraints

The Runtime Engine should avoid asking for information already stored inside AI Architect OS.

---

## Step 3 — Generate Dynamic Session Prompt

Use:

- Session Templates
- Current State
- Curriculum
- AI Manager Rules

to generate one Dynamic Session Prompt.

The prompt becomes the starting point for the AI learning session.

---

## Step 4 — Start AI Session

The AI begins teaching from the reconstructed learning state.

The session should continue naturally without restarting completed topics.

---

## Step 5 — Teach & Mentor

During the session the AI should:

- Teach
- Explain
- Mentor
- Answer questions
- Encourage implementation
- Observe understanding

Teaching should adapt continuously.

---

## Step 6 — Verify Competency

Verification should only occur when necessary.

If competency has already been demonstrated naturally, continue.

Otherwise generate one short practical verification.

---

## Step 7 — Generate Session Update

At the end of the session the AI generates a structured Session Update.

The Session Update summarizes the completed work and determines the next logical continuation.

---

## Step 8 — Update Operating System

The user reviews the generated Session Update.

After confirmation, AI Architect OS is updated.

The Operating System becomes the new source of truth.

---

## Step 9 — Prepare Next Session

The updated Operating System becomes the input for the next Runtime Cycle.

Learning continuity is preserved automatically.

---

## Runtime Principles

The Runtime Engine should always:

- Continue from the latest confirmed state.
- Minimize manual work.
- Preserve curriculum order.
- Adapt to available time.
- Prioritize competency over time spent.
- Generate useful Session Updates.

---

## Constraints

The Runtime Engine should never:

- Restart completed topics.
- Ignore unfinished work.
- Lose learning continuity.
- Depend on one AI model.
- Require unnecessary user input.

---

## Success Criteria

A successful Runtime Engine should allow any compatible AI to immediately continue the learning journey while requiring minimal setup from the user.
